# Validation Field Guide

Status: EXPERIMENTAL

## Goal

Collect decision-grade evidence with minimal outreach and no invented traction.

## Participant groups

### Comprehension group

Ten independent people who have not received a technical explanation before opening the preview.

### Practitioner group

At least five people in one of these categories:

- developer or architect operating agent workflows;
- product owner responsible for automated actions;
- operations or support practitioner handling failed outcomes;
- compliance, audit, security, or trust practitioner.

## Neutral invitation

Use a neutral request: review the preview, complete the local test, download the anonymous JSON, and return that file. Do not prime participants with the correct interpretation.

## Interview discipline

Separate these signals:

1. **Problem recognition** — the workflow can report completion too early.
2. **Integration interest** — the person would test a reusable sandbox connector.
3. **Commercial signal** — a paid value unit and budget range are identified.
4. **Commitment** — a concrete sandbox evaluation is agreed through a separate consensual follow-up.

General enthusiasm is not a commitment. A sandbox commitment is not a purchase. A budget range is not revenue.

## Low-prospecting distribution order

1. Existing trusted contacts able to review a workflow.
2. Relevant developer or agent communities where experimental project feedback is allowed.
3. Public technical post showing the four outcomes and asking for structured feedback.
4. Inbound visitors from protocol documentation and demonstrations.

Do not use high-volume cold outreach for this sprint.

## Decision rule

After the minimum sample is reached, choose exactly one:

- `PROCEED` — select and build the first reusable sandbox connector;
- `REFINE` — revise the packet and repeat comprehension testing;
- `NARROW` — focus on a more urgent action domain;
- `STOP` — evidence does not justify further investment.
