# Distribution and Adoption

## Principle

Distribution must be designed into the product. The default strategy is product-led and ecosystem-led, not high-volume outbound prospecting.

## Channels

- GitHub repository and public specifications;
- npm and PyPI packages after package gates;
- public verifier and tamper demonstrations;
- MCP and agent developer ecosystems;
- ACP/UCP/AP2 compatibility content;
- reference integrations and examples;
- technical launch communities;
- security, payments, commerce, and compliance communities;
- first-party SEO and AI-first documentation;
- standards and research participation after implementation credibility;
- selected design partners for validation, not a consulting pipeline.

## Adoption funnel

```text
Problem education
→ working demo
→ local verification
→ five-minute instrumentation
→ first external evidence
→ retained and re-verified evidence
→ paid assurance
```

## Required assets

- one-command verifier;
- copy-paste SDK quickstarts;
- deterministic examples;
- public compatibility matrix;
- transparent limitations;
- migration guides;
- changelog and semantic versioning;
- responsive documentation and support boundaries.

## Distribution anti-patterns

- launching before the demo explains the value;
- content without first-party technical substance;
- fake community signals;
- relying on a single chat platform;
- publishing many packages before one integration works well;
- accepting custom work that cannot become a reusable connector.
